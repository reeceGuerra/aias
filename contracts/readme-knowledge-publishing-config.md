# Knowledge Publishing Config Contract — Cursor Configuration System (v1.0)

This document defines the canonical contract for knowledge publishing configuration artifacts used by provider-specific knowledge integrations.

It exists to:
- Standardize publishing configuration across knowledge providers (Confluence, Notion, etc.).
- Decouple publishing hierarchy and navigation logic from provider-specific skills.
- Enforce deterministic publishing behavior for artifact sync operations.
- Keep publishing structure auditable and project-customizable.

---

## Purpose

A knowledge publishing config file defines how a knowledge provider organizes and publishes task artifacts. It is project-specific — different companies will have different spaces, hierarchies, and title conventions.

The config is consumed via `config_source` in `aias-config/providers/knowledge-config.md`.

---

## File Location

Publishing config artifacts live in the Referenced Document Directory:

- `aias-config/providers/<provider_id>/confluence-config.md` (canonical, generated by `/aias configure-providers`)

Legacy location (deprecated in v7.5):

- `aias/.skills/<provider-skill>/confluence-config.md`

The active file must be referenced by `config_source` in `aias-config/providers/knowledge-config.md` and listed in `skill_binding.resource_files`.

---

## Mandatory Sections (Order)

Each publishing config file must contain these sections in this exact order:

1. `Space`
2. `Publishing Hierarchy`
3. `TECH Resolution`
4. `Date Resolution`
5. `Navigation Algorithm`
6. `Rules`
7. `Example`

---

## Normative Schema

### Space

Each config must declare, at minimum:

- `space_key` (string, provider space identifier)
- `space_id_resolution` (string, description of how to resolve the numeric ID at runtime)
- `root_parent_page` (string, human-readable name)
- `root_parent_page_id` (string, provider-specific page ID)

Optional:
- `space_url` (string, human-readable URL)
- `root_parent_page_url` (string, human-readable URL)

### Publishing Hierarchy

The config must define a hierarchy template that determines how artifacts are organized under the root page. The template must use parameterized segments.

Required parameters:
- A technology/platform grouping segment
- A time-based grouping segment (year, quarter, or equivalent)
- A task identifier segment

### TECH Resolution

The config must define a deterministic priority-ordered resolution strategy for the technology grouping segment. At minimum:

1. A primary source (e.g., tracker metadata)
2. A fallback source (e.g., workspace metadata)
3. A last-resort strategy (e.g., user prompt — never guess)

The config must include a table of supported technology values with their mapping from each resolution source.

### Date Resolution

The config must define how time-based segments are derived. The resolution must be deterministic and based on a single, unambiguous source (e.g., system date at publish time).

### Navigation Algorithm

The config must define a `findOrCreatePage` (or equivalent) algorithm that:

1. Searches existing pages before creating new ones (duplicate prevention is mandatory).
2. Handles each level of the hierarchy sequentially.
3. Defines title conventions that prevent collisions (scope-aware titles for repeating nodes).
4. Defines behavior for both `created` and `modified` artifact sync statuses.
5. Defines legacy title fallback behavior for backward compatibility.

### Rules

The config must declare operational invariants as an explicit rules list. At minimum:

- Duplicate prevention rule (never create without searching first)
- Runtime ID resolution rule (never hardcode numeric IDs that can be resolved)
- Content format rule (what format to pass to the provider API)
- Title matching rule (case sensitivity, exactness)
- Content completeness rule (publish full content, never truncate)
- Graceful degradation rule (behavior when provider is unreachable)

### Example

The config must include at least one concrete example showing the full hierarchy for a representative task.

---

## Resolution Rules (Fail-Fast)

When resolving publishing configuration:

- If config file is missing: abort dependent publishing operation.
- If mandatory sections are absent: abort dependent publishing operation.
- If space key or root page cannot be resolved at runtime: abort dependent publishing operation.
- If navigation algorithm cannot resolve or create a page: log warning, keep artifact sync status unchanged, do not block command execution.

Error response must include:

- provider
- config_source path
- reason code

## Conformance Checklist

| Check | Pass/Fail |
|---|---|
| Mandatory section order present | ☐ |
| Space key and root page declared | ☐ |
| Publishing hierarchy template defined | ☐ |
| TECH resolution has priority-ordered sources | ☐ |
| Date resolution is deterministic | ☐ |
| Navigation algorithm prevents duplicates | ☐ |
| Title convention handles collisions | ☐ |
| Rules section declares all mandatory invariants | ☐ |
| Example section present | ☐ |
