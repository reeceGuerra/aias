# Fix (Debug Analysis) — v3

## 1. Identity

**Command Type:** Operative — Procedural / Execution

You are generating a **debug analysis and fix plan** to guide implementation of bug fixes.
This command is responsible for transforming raw debugging data from `@debug` (including any code/diffs suggested in chat) into a structured `analysis.fix.md` artifact in the task directory. That artifact is **context for @dev in another chat**, which can execute the fix. Optional: consume `report.issue.md` from TASK_DIR as repro context; include a "Validation" section.

**Skills referenced:** `rho-aias`, `technical-writing`.

---

## 2. Invocation / Usage

Invocation:
- `/fix`

Usage notes:
- This command is intended to be used **after** `@debug` mode has generated raw debugging data.
- It MAY use a bug report generated by `/issue` as additional context.
- It must not be invoked as a first step when debugging information is incomplete.
- The output will be saved to an `analysis.fix.md` file.

---

## 3. Inputs

This command may use **only** the following inputs:
- Chat context explicitly provided by the user
- Output from `@debug` mode (raw debugging data: error mechanisms, hypotheses, log analysis, etc.)
- Bug report from `/issue` (optional, loaded from TASK_DIR via **rho-aias** skill: `report.issue.md`)
- Logs, diffs, or notes explicitly pasted by the user

Rules:
- All inputs must be explicit.
- If required information is missing, the command must request it before producing output.
- If a bug report exists in TASK_DIR, load it as additional context.

---

## 4. Output Contract (Format)

### Gate: Artifact Preview

**Type:** Confirmation
**Fires:** Before writing `analysis.fix.md` to TASK_DIR.
**Skippable:** No.

**Context output:**
Present artifact summary in chat:
- Artifact: `analysis.fix.md`
- Target: TASK_DIR path
- Mechanism count (confirmed vs discarded)
- Solution count (viable)

**AskQuestion:**
- **Runtime compatibility:** If `AskQuestion` is unavailable, use the Text Gate Protocol from `readme-commands.md` with the same prompt, option ids, labels, and `allow_multiple` semantics.
- **Prompt:** "Write fix analysis to <TASK_DIR>?"
- **Options:**
  - `write`: "Write artifact to TASK_DIR"
  - `adjust`: "Adjust content before writing"
- **allow_multiple:** false

**On response:**
- `write` → Write artifact to TASK_DIR, proceed to End-of-Response Confirmation
- `adjust` → Apply corrections, return to context output and re-present gate

**Anti-bypass:** Inherits Gate Invocation Protocol. No additional rules.

FILE OUTPUT CONTRACT (must follow)
- Follow the **rho-aias** skill loading protocol Phase 0 to resolve TASK_DIR.
- If TASK_DIR is set: write `analysis.fix.md` to TASK_DIR.
- If TASK_DIR is not set: create a new directory `<resolved_tasks_dir>/<Name>/` (default: ~/.cursor/plans/) using a kebab-case name derived from the fix title, and write `analysis.fix.md` there.
- Create `status.md` if it does not exist (profile: `bugfix`).
- The content may include guidance and, if @debug provided it in chat, code or suggested patches for @dev to apply.

STATUS UPDATE (Phase 5)

- Run Phase 5c: sync non-synced artifacts to resolved knowledge provider. Phase 5c always publishes — it is NOT conditioned by plan classification (see **rho-aias** skill § Phase 5c).

END-OF-RESPONSE CONFIRMATION (must follow)
- After writing the fix analysis file, you MUST print a final line in the chat response (not in the file) exactly in this format:
  Saved fix to: <absolute_path>
- <absolute_path> must resolve to the fully expanded absolute path.

---

## 5. Content Rules (Semantics)

- Output must be in **English**.
- Apply the **technical-writing** skill patterns: Root Cause Description, Risk Articulation, Conciseness.
- Do **NOT** invent information.
- Use **ONLY** the provided inputs from `@debug` mode.
- If information is missing, leave the section empty or include a short placeholder comment.
- Focus on **root-cause isolation** and **minimal-risk remediation guidance**.
- Do **NOT** include implementation code or full file contents.
- Treat proposed solutions as hypotheses until validated.
- Prioritize evidence over guesses.

---

## 6. Output Structure (Template)

The fix analysis file must follow this exact structure:

```markdown
## Fix Analysis: <clear, concise title>

### 1) Problem Restatement (1–3 lines)
<!-- Summarize the bug in your own words using QA report facts only -->

---

### 2) Likely Error Mechanisms (ranked, max 5)

For each mechanism:
- **Why it's plausible**: Tie to repro steps, expected/actual, logs, and/or code anchors
- **Supporting evidence**: Explicit references
- **What would falsify it**: Conditions that would rule this out
- **Minimal validation step**: Fast, low-risk validation (may include targeted logging/instrumentation suggestions)

#### Mechanism 1: <mechanism name>
<!-- Details above -->

#### Mechanism 2: <mechanism name>
<!-- Details above -->

<!-- Continue for up to 5 mechanisms -->

---

### 3) Recommended Solutions (ranked, max 5)

For each solution:
- **Which mechanism(s) it addresses**: Link to mechanisms above
- **Why it's recommended**: Impact + risk assessment
- **Validation plan**: How to confirm the fix works
- **Risk notes**: Potential side effects + rollback/mitigation options

#### Solution 1: <solution name>
<!-- Details above -->

#### Solution 2: <solution name>
<!-- Details above -->

<!-- Continue for up to 5 solutions -->

---

### 4) If evidence is insufficient
- **What is missing**: State exactly what information is needed
- **Next best signal to collect**: Log lines, correlation IDs, reproduction matrix, etc.
- **Safe mitigation**: If available, provide a temporary workaround
```

---

## 7. Non-Goals / Forbidden Actions

This command must **NOT**:
- Perform debugging or analysis (that's `@debug` mode's job)
- Invent mechanisms not grounded in @debug output
- Infer missing context
- Write or modify code (except writing the fix file to TASK_DIR)
- Touch the repository (no files or dirs in the repo)
- Write artifacts outside TASK_DIR
- Propose refactors or architectural changes
- Execute commands or scripts
- Assume causality from log order alone
- Recommend changes without validation plans
